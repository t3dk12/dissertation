# Dissertation

This repository is my Part C lithium project workspace, built from I-ReaxFF and then cleaned for my specific run workflow.

## Project Goal

- Train a lithium-focused IRFF/MPNN model from OUTCAR-derived trajectories.
- Keep the workflow reproducible (setup -> dataset build -> training -> conversion/check).
- Keep only the files needed for this project run.

## How This Differs from Base I-ReaxFF

This started from a pure clone of:

`https://github.com/fenggo/I-ReaxFF.git`

Then it was updated in stages:

1. Added project run scripts and setup scripts.
2. Added raw standard force fields (`44_36`, `179_3`, `2024vD`).
3. Pruned non-essential upstream folders from `I-ReaxFF` (`doc`, `examples`, `test`, most `tools`).
4. Applied working code edits in:
   - `I-ReaxFF/irff/reax.py`
   - `I-ReaxFF/irff/link.py`
   - `I-ReaxFF/irff/mpnn.py`
   - `I-ReaxFF/tools/json_to_ffield.py`
   - Added `I-ReaxFF/tools/ffield_to_json_full.py`
   - Added `I-ReaxFF/tools/build_li_from_sources.py`
5. Added converted JSON potentials and verification outputs.
6. Set active run potential to root `ffield.json` (from 2024vD Li-only).

## Repository Layout

- `I-ReaxFF/` editable project copy used for this workflow.
- `run_scripts/` project scripts:
  - `build_dataset_duplicate.py`
  - `train_without_force_Li_NN.py`

- `verify/` conversion verification outputs.

## How to Run

Run from repository root.

### Option A: One Command (Recommended)

```bash
bash setup_and_run.sh
```

This will:
1. Run environment setup (`setup.sh`).
2. Build the dataset if needed.
3. Run training if convergence is not already detected in `training.log`.

### Option B: Step-by-Step

#### 1) Set up Environment

```bash
bash setup.sh
conda activate li-model
```

#### 2) Place Dataset

Put your OneDrive OUTCAR folders in:

`data/Li-metal_OUTCARs/`

#### 3) Build Dataset from OUTCAR

```bash
python run_scripts/build_dataset_duplicate.py \
  --input_dir data/Li-metal_OUTCARs \
  --output_dir data/traj \
  --ffield ffield.json
```

Expected outputs:

- `data/traj/train/*.traj`
- `data/traj/test/*.traj`
- `data/traj/dft_summary.csv`

#### 4) Train

```bash
python run_scripts/train_without_force_Li_NN.py \
  --train_dir data/traj/train \
  --ffield ffield.json
```

Example full command used in this workflow:

```bash
python run_scripts/train_without_force_Li_NN.py \
  --train_dir data/traj/train \
  --ffield ffield.json \
  --s 10000 --lr 0.0001 --pr 200 --writelib 1000 --batch 50 \
  --bo 0 --h 0 --a 0 --t 0 --f 0 --vdw 1
```

## Planned Stage: TensorFlow 1 -> TensorFlow 2 Migration

This is not part of the current run stage and is planned for a later phase.

Likely migration approach:

1. Identify TF1-only usage across `I-ReaxFF` (placeholders, sessions, `tf.compat.v1` paths).
2. Migrate module-by-module to TF2-compatible patterns while preserving current behaviour.
3. Keep training outputs comparable during migration (loss trend, penalties, and convergence behaviour).
4. Validate converted scripts against the same dataset and `ffield.json` before replacing the current run path.

Basic multi-GPU plan for that stage:

1. Keep the current physics terms and loss definitions, but replace TF1 session/placeholder training loops with TF2 training steps.
2. Introduce `tf.distribute.MirroredStrategy` for single-node multi-GPU training.
3. Move input loading to a TF2 data pipeline and split global batch size across available GPUs.
4. Keep checkpoint/writelib behaviour consistent so existing run monitoring still works.
5. Compare single-GPU and multi-GPU runs on the same dataset to confirm equivalent convergence behaviour.

Expected result of this stage:

- Reduced dependency on TF1 compatibility mode.
- Clearer long-term maintenance path for training scripts.
- A tested TF2 workflow that can be used for future ReaxFF-NN work.
